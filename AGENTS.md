## Overview

This project uses **Mixpanel** for product analytics. All tracking is implemented
through a thin wrapper module so that:

- The Mixpanel SDK is only imported from one place.
- Tracking calls are safe to call before initialization (they are no-ops).
- Token configuration is centralised in a single environment variable.

---

## Import Rules

**Always** import tracking helpers from the wrapper module:

```typescript
import { track, identify, reset } from "@/lib/mixpanel";
```

**Never** import `mixpanel-browser` directly in application code:

```typescript
// BAD — do not do this
import mixpanel from "mixpanel-browser";
mixpanel.track("event");
```

Direct SDK usage bypasses the initialisation guard and makes it harder to
mock tracking in tests.

## Event Naming Rules

Events follow the **object_verb** pattern in `snake_case`:

| Pattern | Example |
|---------|---------|
| `<object>_<verb>` | `button_clicked`, `form_submitted`, `page_viewed` |

Rules:
- Use `snake_case` only — no camelCase, PascalCase, or spaces.
- Be specific about the object: `signup_form_submitted`, not `form_submitted`.
- Use past tense for the verb: `_clicked`, `_submitted`, `_viewed`, `_completed`.
- Keep names stable — renaming an event breaks historical data.

## When to Track

Track user actions that signal meaningful intent or progress:

- Completing a key workflow step (signup, purchase, onboarding milestone).
- Clicking a primary CTA.
- Viewing a page or screen that indicates user interest.
- Errors that affect the user experience.
- Feature adoption moments.

## When NOT to Track

- Internal admin or developer actions.
- Background system events with no user intent.
- PII (names, emails, passwords, payment details) — use anonymous IDs.
- Mouseover or hover events (too noisy).
- Every keystroke in a form (use `form_submitted` instead).

## Super Properties

Super properties are registered once and attached to every subsequent event
automatically. Do not re-register them on every track call.

```typescript
import { registerSuperProperties } from "@/lib/mixpanel";

registerSuperProperties({
  // register after you know the value, e.g. after auth
});
```

Registered super properties for this project:

- `user_role`: from user.role field
- `team_plan`: from team.planName or 'Free'
- `subscription_status`: from team.subscriptionStatus

## Identity Management

Mixpanel distinguishes anonymous visitors from identified users.

| Action | When | File |
|--------|------|------|
| `identify(userId)` | successful login and app initialization | `app/(login)/actions.ts` |
| `reset()` | user logout and session end | `app/(login)/actions.ts` |

Rules:
- Call `identify` **once** per session, immediately after the user authenticates.
- Pass a **stable, non-PII** user ID (e.g. a database UUID). Never pass email.
- Call `reset` on logout so the next user on the same device gets a fresh
  anonymous profile.
- Do not call `identify` on every page load — it creates unnecessary API calls.

## Adding New Events

1. Add the tracking call in the relevant source file using the wrapper:
   ```typescript
   import { track } from "@/lib/mixpanel";
   track("object_verb", { property: value });
   ```
2. Document the event in this file under **Tracking Plan** (or re-run the
   GitHub App to regenerate AGENTS.md).
3. Add a Playwright test under `tests/mixpanel/` that intercepts the
   Mixpanel network request and asserts the event name and key properties.
4. Open a PR — the CI workflow will run the Playwright tracking tests
   automatically on branches prefixed with `mixpanel/`.
